# 执行计划

## 总体规则

项目分为两个板块：

1. 生成
2. 移调

当前只优先做第一板块“生成”。

第一板块再分成两个版本：

- v1：无 key 输入
- v2：有 key 输入

执行顺序必须严格按这个结构推进，不要一开始同时做所有内容。

---

## Phase 0：锁定任务边界

### 目标
明确当前到底在做什么。

### 要做的事
- 明确整个项目分为两个板块：
  - 生成
  - 移调
- 明确当前只做第一板块
- 明确第一板块的两个版本：
  - v1：text -> sample
  - v2：text + key -> sample
- 明确当前不做的内容：
  - reference audio
  - 插件实现
  - 完整虚拟乐器
  - 长 loop
  - 复杂 UI

### 需要返回的结果
1. 当前任务定义
2. 两个板块的边界
3. v1 与 v2 的明确定义
4. 不做内容清单

---

## Phase 1：选择底座模型

### 目标
选择一个适合微调的开源底座。

### 要做的事
- 搜索 2 到 3 个候选模型
- 比较它们是否支持：
  - 短音频生成
  - 微调
  - 本地运行
- 选择一个主底座，一个备用底座

### 需要返回的结果
1. 候选模型列表
2. 每个模型的优缺点
3. 最终推荐
4. 为什么这个底座适合当前项目

---

## Phase 2：设计数据格式

### 目标
定义数据应该长什么样。

### v1 数据格式
- audio
- text

### v2 数据格式
- audio
- text
- key

### 要做的事
- 定义元数据 schema
- 设计文件结构
- 明确 key 字段在 v2 中如何表示
- 明确当前不引入多余字段

### 需要返回的结果
1. JSON 或 CSV 样例
2. 文件夹结构
3. 字段说明
4. 哪些字段属于 v1，哪些属于 v2

---

## Phase 3：寻找数据来源

### 目标
找到适合训练短音频 sample 生成的数据。

### 要做的事
- 搜索可用数据集或素材来源
- 判断哪些来源适合：
  - 时长短
  - 可用于训练
  - 尽量主音高稳定
- 剔除明显不适合的来源

### 需要返回的结果
1. 候选来源列表
2. 每个来源是否可用
3. 最终推荐的数据来源方案
4. 预计需要多少人工清洗工作

---

## Phase 4：预处理和清洗

### 目标
把所有数据统一成可训练格式。

### 要做的事
- 统一采样率
- 统一声道设置
- 统一时长裁剪规则
- 做音量归一化
- 去掉无效样本
- 检查静音和损坏文件

### 需要返回的结果
1. 预处理规则
2. 预处理脚本方案
3. 清洗后的统计结果
4. 异常样本说明

---

## Phase 5：设计 prompt 规范

### 目标
让文本输入尽量统一。

### 要做的事
- 设计 v1 prompt 模板
- 设计 v2 prompt 模板
- 控制词汇风格一致
- 避免描述过度自由

### 示例
v1:
- warm pluck
- dark bass one-shot

v2:
- warm pluck in C
- dark bass one-shot in A

### 需要返回的结果
1. prompt 模板规则
2. 词汇表建议
3. 20 个示例 prompt
4. v1 / v2 prompt 的差异说明

---

## Phase 6：探索性数据分析

### 目标
确认数据集是否真的可用。

### 要做的事
- 统计样本数量
- 统计时长分布
- 统计文本词频
- 如果有 key，统计 key 分布
- 检查重复样本
- 检查严重不平衡问题

### 需要返回的结果
1. 数据统计表
2. 主要问题列表
3. 是否可以进入训练
4. 还需不需要补数据

---

## Phase 7：训练 v1 baseline

### 目标
先做出一个最简单可运行的模型。

### 要做的事
- 用 v1 数据训练 text-only 模型
- 不加入 key 输入
- 不加入额外 key loss
- 保存 checkpoint 和生成样例

### 需要返回的结果
1. 训练配置
2. loss 曲线
3. 生成样例
4. 成功和失败案例分析

---

## Phase 8：评测 v1

### 目标
判断 v1 是否足够稳定。

### 要做的事
- 做主观听感检查
- 做文本对齐检查
- 判断输出是否适合作为短 sample

### 需要返回的结果
1. 评测方法
2. 成功样本
3. 失败样本
4. 是否可以进入 v2

---

## Phase 9：训练 v2（加入 key 输入）

### 目标
在 v1 基础上加入 key 控制。

### 要做的事
- 把 key 加进输入
- 先训练 text + key 模型
- 暂时可以先不加额外 key loss
- 观察模型是否已经学到部分 key 控制

### 需要返回的结果
1. v2 输入格式
2. 训练配置
3. 多个 key 下的生成样例
4. 初步 controllability 分析

---

## Phase 10：设计并加入 L_key

### 目标
让模型更明确地学会 key 控制。

### 要做的事
- 定义总 loss：
  - L = L_gen + λL_key
- 选择 L_key 的实现方式
- 如果音高函数不可微，不直接把它接进训练图
- 优先考虑训练一个可微代理 head：
  - key classifier
  - 或 pitch regressor

### 推荐顺序
1. 先尝试离散分类
2. 再决定是否需要其他形式

### 需要返回的结果
1. L_key 定义
2. 代理 head 设计
3. 可微性说明
4. 为什么选择这种方案

---

## Phase 11：评测 v2

### 目标
验证 key 控制是否有效。

### 要做的事
比较以下几组：

- v1：text-only
- v2：text + key
- v2：text + key + L_key

重点检查：
- 生成质量
- 文本对齐
- key 准确性

### 需要返回的结果
1. 对比表格
2. 关键指标
3. 典型成功案例
4. 典型失败案例

---

## Phase 12：做消融实验总结

### 目标
把结果整理成可用于面试的证据。

### 要做的事
- 汇总 v1 和 v2 差异
- 汇总是否加入 L_key 的差异
- 分析 key 输入和 key loss 的作用

### 需要返回的结果
1. 消融表
2. 关键结论
3. 每个设计选择带来的影响
4. 面试可用的简洁表述

---

## Phase 13：打包第一板块 demo

### 目标
准备展示生成模块。

### 要做的事
- 选一批 prompt
- 生成一批代表性结果
- 如果已有 v2，展示不同 key 下的输出
- 准备简单的说明文案

### 需要返回的结果
1. demo prompt 列表
2. demo 音频列表
3. 每个例子的说明
4. 推荐展示顺序

---

## Phase 14：规划第二板块“移调”

### 目标
只做规划，不立即全面实现。

### 要做的事
- 定义第二板块的输入输出
- 明确它和第一板块的接口
- 明确后续要不要直接用现成算法
- 明确这个板块的评测方式

### 需要返回的结果
1. 第二板块定义
2. 和第一板块的连接方式
3. 可用现成方法列表
4. 后续开发建议

## 每一步统一汇报格式

每个阶段结束后，必须返回：

1. 做了什么  
2. 找到了什么  
3. 当前问题是什么  
4. 下一步具体做什么

不要返回空泛总结。  
必须尽量给出具体产物：

- 配置
- 表格
- 路径
- 数据统计
- 代码改动
- 生成结果

---

## 优先级规则

如果必须取舍，优先顺序如下：

1. 明确两块结构
2. 跑通第一板块 v1
3. 跑通第一板块 v2
4. 做清楚评测
5. 设计第二板块

不要为了功能多而牺牲清晰度。

---

## 一句话执行摘要

整个项目分成“生成 + 移调”两块。  
当前先做生成模块，并严格按下面顺序推进：

- v1：text -> sample
- v2：text + key -> sample
- 再考虑加入 L_key
